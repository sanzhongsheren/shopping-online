# 实验十二 任务3：面向对象设计原则论述

> **项目**：商品网上交易系统（shopping-online）
> **技术栈**：Node.js + Express + MySQL
> **论述原则**：里氏代换原则、单一职责原则、开闭原则、迪米特法则、依赖倒转原则、合成复用原则

---

## 一、单一职责原则（Single Responsibility Principle, SRP）

### 定义

一个类或模块应该**有且仅有一个引起它变化的原因**。换句话说，一个模块只负责完成一项职责。

### 项目实践

#### 正面案例：`utils/response.js` —— 专一的响应格式化工具

[backend/utils/response.js](backend/utils/response.js) 的职责极为单一——只负责统一 API 响应格式，所有控制器都通过它返回标准化 JSON：

```js
module.exports = {
    success(res, data = null, message = "success", code = 200) {
      return res.status(code).json({ code, message, data });
    },
    error(res, message = "服务器内部错误", code = 500) {
      return res.status(code).json({ code, message });
    },
    notFound(res, message = "资源不存在") {
      return res.status(404).json({ code: 404, message });
    },
    badRequest(res, message = "请求参数错误") {
      return res.status(400).json({ code: 400, message });
    }
};
```

当需要调整响应格式时，只需修改此文件，无需改动任何控制器。这正是 SRP 的价值——隔离变化的影响范围。

#### 正面案例：路由与控制器分离

在 [backend/routes/orders.js](backend/routes/orders.js) 中，路由文件**仅负责 URL 到控制器的映射**，不含任何业务逻辑：

```js
router.post("/", authenticate, authorizeRoles("buyer"), orderController.createOrder);
router.get("/my", authenticate, authorizeRoles("buyer"), orderController.getMyOrders);
router.get("/seller", authenticate, authorizeRoles("seller"), orderController.getSellerOrders);
```

新增路由时只需添加一行映射，不涉及逻辑修改。

#### 反面案例：`userController.updateProfile` 越权直接操作数据库

[backend/controllers/userController.js:108-119](backend/controllers/userController.js#L108-L119) 中，控制器绕过了 User 模型，直接写 SQL：

```js
exports.updateProfile = async (req, res) => {
  try {
    const { email, phone, real_name } = req.body;
    const db = require("../config/db");                    // ← 绕过模型层
    const sql = "UPDATE users SET email = ?, phone = ?, real_name = ? WHERE user_id = ?";
    await db.query(sql, [email, phone, real_name, req.user.userId]);
    success(res, null, "个人信息已更新");
  } catch (err) { ... }
};
```

控制器本应只负责请求参数的接收与校验，却在这里承担了**数据访问**的职责。应改为调用 `User.updateProfile()` 将数据访问委托给模型层。

#### 反面案例：前端 `api.js` 的单体设计

[frontend/js/api.js](frontend/js/api.js) 将所有 API 模块（用户、商品、购物车、订单、审核、留言、收藏）集中在一个 `API` 对象中（340+ 行）。当任何一个模块的接口变更，都需要修改此文件。更好的做法是拆分为 `userApi.js`、`productApi.js`、`orderApi.js` 等独立模块。

## 二、开闭原则（Open/Closed Principle, OCP）

### 定义

软件实体（模块、类、函数）应该**对扩展开放，对修改关闭**。即新增功能应通过扩展实现，而非修改已有代码。

### 项目实践

#### 正面案例：Express 中间件链——最经典的 OCP 实践

在 [backend/app.js](backend/app.js) 中，全局中间件通过 `app.use()` 层层叠加：

```js
app.use(cors());
app.use(express.json());
app.use(express.urlencoded({ extended: true }));
app.use("/api/products", require("./routes/products"));
app.use("/api/users", require("./routes/users"));
// ...
app.use((err, req, res, next) => { ... });
```

若要添加日志记录、限流、压缩等新功能，只需插入新的 `app.use(...)` 行即可，**无需修改任何已有中间件或路由代码**。

#### 正面案例：`authorizeRoles` 的开放参数设计

[backend/middleware/auth.js:22-29](backend/middleware/auth.js#L22-L29) 使用 rest 参数接受任意数量的角色：

```js
exports.authorizeRoles = (...roles) => {
  return (req, res, next) => {
    if (!req.user || !roles.includes(req.user.role)) {
      return res.status(403).json({ code: 403, message: "权限不足" });
    }
    next();
  };
};
```

未来即使引入 `superadmin`、`moderator` 等新角色，此函数完全不需任何修改——它在设计之初就对角色扩展开放。

#### 反面案例：订单模型的硬编码状态转换

[backend/models/order.js](backend/models/order.js) 中，每个状态转换函数都硬编码了前置状态和目标状态：

```js
// payOrder → 硬编码 status = 'pending_payment' → 'paid'
async function payOrder(orderId, buyerId) {
    const sql = "UPDATE orders SET status = 'paid', payment_time = NOW()
                 WHERE order_id = ? AND buyer_id = ? AND status = 'pending_payment'";
    ...
}

// shipOrder → 硬编码 status = 'paid' → 'shipped'
async function shipOrder(orderId, sellerId) {
    const sql = `UPDATE orders o SET o.status = 'shipped', o.shipping_time = NOW()
                 WHERE o.order_id = ? AND o.status = 'paid'`;
    ...
}
```

如果需要新增"退款中（refunding）"状态，必须修改现有 `payOrder`、`cancelOrder` 等多个函数的内部逻辑。**改进方案**：引入状态机模式，将状态与合法转换定义在配置中，由通用转换函数统一处理，新状态只需声明配置即可。

## 三、里氏代换原则（Liskov Substitution Principle, LSP）

### 定义

子类型必须能够**完全替代**它们的父类型，而程序的行为不发生改变。即：程序中使用父类引用的地方，替换为子类对象后应依然正确运行。

### 项目实践

本项目未使用传统的类继承（无 `class` 或 `extends`），但 LSP 的思想在 **接口契约** 层面同样适用。

#### 正面案例：中间件函数的可替换性

[backend/middleware/auth.js](backend/middleware/auth.js) 中，`authenticate` 与 `authorizeRoles()` 的返回值具有完全一致的函数签名 `(req, res, next) => void`：

```js
exports.authenticate = (req, res, next) => {
  // 验证 token，设置 req.user
  next();
};

exports.authorizeRoles = (...roles) => {
  return (req, res, next) => {
    // 检查角色权限
    next();
  };
};
```

`authenticate` 和 `authorizeRoles("buyer")` 在路由中可以**任意替换、组合**，路由行为保持正确：

```js
// 三段中间件可随意替换、重排，系统行为不变
router.post("/", authenticate, authorizeRoles("buyer"), orderController.createOrder);
```

这正是 LSP 在函数式编程风格中的体现——行为相同的接口（相同签名）可互相替换。

#### 正面案例：控制器函数的统一接口

项目中所有控制器函数都遵循 `(req, res) => Promise<void>` 的签名约定。这意味着 Express 路由可以将任何控制器函数赋给任何路由，系统均能正确处理。例如将 `productController.approveProduct` 替换到订单路由中（逻辑虽然不对，但调用行为正常），这说明 LSP 在接口层面得到了遵守。

#### 可改进之处

项目未使用类继承，因此不存在传统意义上 LSP 被违反的场景（如子类重写方法后改变了父类契约）。但这也意味着没有利用 LSP 的优势——例如没有定义 `BaseController` 来统一错误处理、参数校验，导致每个控制器中都有大量重复的 `try-catch` 模板代码。如果引入继承体系，LSP 将发挥更大价值。

## 四、迪米特法则（Law of Demeter, LoD）

### 定义

一个对象应该**对其他对象有尽可能少的了解**，只与"直接朋友"通信，不与"陌生人"打交道。也称为最少知识原则。通俗地说：**不要和陌生人说话**。

### 项目实践

#### 正面案例：控制器只与直接依赖通信

[backend/controllers/orderController.js](backend/controllers/orderController.js) 的依赖关系非常干净：

```js
const Order = require("../models/order");           // 直接朋友：Order 模型
const { success, error, badRequest, notFound } = require("../utils/response");  // 直接朋友：响应工具

exports.getMyOrders = async (req, res) => {
  try {
    const orders = await Order.getOrdersByBuyer(req.user.userId);
    success(res, { orders }, "订单列表获取成功");
  } catch (err) { ... }
};
```

控制器只和 `Order` 模型、`response` 工具这两个"直接朋友"通信，**没有跨层访问** `Order` 内部的实现细节（如数据库连接、SQL 语句）。整个项目中不存在 `a.b.c.d` 这种"火车式"链式调用。

#### 正面案例：模型只与 `db` 通信

[backend/models/order.js](backend/models/order.js) 中，模型层只引入 `db` 一个依赖：

```js
const db = require("../config/db");

async function getOrdersByBuyer(buyerId) {
  const sql = "SELECT ... FROM orders o WHERE o.buyer_id = ? ...";
  const [rows] = await db.query(sql, [buyerId]);
  return rows;
}
```

模型不依赖其他模型，不依赖控制器，不依赖路由。它只知道数据库连接——这是最少的"知识"。

#### 反面案例：`productController.createProduct` 同时与两个模型通信

[backend/controllers/productController.js:60-84](backend/controllers/productController.js#L60-L84) 中：

```js
const Product = require("../models/product");
const Audit = require("../models/audit");

exports.createProduct = async (req, res) => {
  try {
    const productId = await Product.createProduct({ ... });
    await Audit.createAudit('product_publish', 'product', productId, req.user.userId);
    ...
  }
};
```

控制器同时"认识" `Product` 和 `Audit` 两个模型，还编排了它们之间的协作流程（先创建、再提交审核）。**改进方案**：引入服务层（如 `ProductService`），将"创建商品+提交审核"封装为原子操作，控制器只与一个 Service 通信。

#### 反面案例：绕过模型层的 `userController.updateProfile`

如 SRP 部分所述，`updateProfile` 直接访问 `db` 对象，使得控制器额外"知道"了数据库驱动和表结构，违反了迪米特法则。

## 五、依赖倒转原则（Dependency Inversion Principle, DIP）

### 定义

- **高层模块不应依赖低层模块**，两者都应依赖抽象（接口）
- **抽象不应依赖细节**，细节应依赖抽象

简而言之：面向接口编程，而非面向实现编程。

### 项目实践

#### 正面案例：控制器依赖模型接口

[backend/controllers/orderController.js](backend/controllers/orderController.js) 中：

```js
const Order = require("../models/order");

exports.createOrder = async (req, res) => {
  const result = await Order.createOrder(req.user.userId, addressId, paymentMethod, items);
  ...
};
```

控制器依赖的是 `Order.createOrder()` 这个**函数接口**，而非其内部如何实现（是调用 MySQL 还是 MongoDB、是单步还是事务），控制器一概不知。这就是"依赖抽象"的体现。

#### 正面案例：路由依赖控制器接口

[backend/routes/orders.js](backend/routes/orders.js) 中：

```js
const orderController = require("../controllers/orderController");
router.post("/", authenticate, authorizeRoles("buyer"), orderController.createOrder);
```

路由只知道 `orderController.createOrder` 是一个 `(req, res) => Promise<void>` 的函数，完全不知道其内部逻辑。

#### 反面案例：模型层强依赖具体数据库实现

[backend/config/db.js](backend/config/db.js) 直接导出 `mysql2/promise` 的连接池：

```js
const mysql = require("mysql2/promise");
const pool = mysql.createPool({ host: ..., port: ..., user: ..., database: ... });
module.exports = pool;
```

而所有模型都依赖这个**具体实现**：

```js
const db = require("../config/db");  // 每个模型都硬依赖 mysql2
```

如果未来要从 MySQL 迁移到 PostgreSQL，**所有模型文件都要修改**。**改进方案**：引入 Repository 抽象层，定义统一的数据访问接口（如 `findById`、`save`、`update`），具体实现由 MySQL Repository 或 PostgreSQL Repository 提供，模型只依赖接口。

#### 反面案例：`userController.updateProfile` 完全绕过了抽象

同 SRP 部分所述，该函数直接依赖 `db.query()` 和原始 SQL，跳过了模型抽象层，是 DIP 违反的典型案例。

## 六、合成复用原则（Composite/Aggregate Reuse Principle, CARP）

### 定义

**优先使用对象组合（composition）而非类继承（inheritance）来实现复用**。继承是静态的、编译期确定的，而组合是动态的、运行时可替换的，更加灵活。

### 项目实践

#### 正面案例：Express 中间件链 —— 组合的典范

[backend/routes/orders.js](backend/routes/orders.js) 中，每个路由都是**功能模块的水平组合**：

```js
// 买家路由：组合 authenticate + authorizeRoles("buyer") + createOrder
router.post("/", authenticate, authorizeRoles("buyer"), orderController.createOrder);

// 卖家路由：组合 authenticate + authorizeRoles("seller") + shipOrder
router.put("/:id/ship", authenticate, authorizeRoles("seller"), orderController.shipOrder);

// 管理员路由：组合 authenticate + authorizeRoles("admin") + approveProduct
router.put("/:id/approve", authenticate, authorizeRoles("admin"), productController.approveProduct);
```

不同的中间件组合产生不同的权限行为，**无需为每种角色创建继承子类**。这比传统的 `BuyerRoute extends BaseRoute` 继承方式灵活得多——组合可以在运行时任意编排。

#### 正面案例：`authorizeRoles` 的高阶函数模式

[backend/middleware/auth.js:22](backend/middleware/auth.js#L22) 使用工厂函数动态生成中间件：

```js
exports.authorizeRoles = (...roles) => {
  return (req, res, next) => { ... };
};
```

这种方式通过**组合参数**来生成不同的行为，**避免了为 `Seller`、`Buyer`、`Admin` 各建一个中间件子类**。使用时 `authorizeRoles("seller")` 和 `authorizeRoles("admin")` 的差异仅由传入的参数决定，复用同一段核心逻辑。

#### 正面案例：模型模块独立，零继承

所有模型文件均为独立的 Node.js 模块，导出纯函数，不存在 `BaseModel` 父类：

| 模型文件 | 导出 |
|---|---|
| [backend/models/user.js](backend/models/user.js) | `{ createUser, findByUsername, findById }` |
| [backend/models/product.js](backend/models/product.js) | `{ getProductList, getProductById, createProduct, ... }` |
| [backend/models/order.js](backend/models/order.js) | `{ createOrder, getOrdersByBuyer, payOrder, ... }` |
| [backend/models/cart.js](backend/models/cart.js) | `{ getCartByUserId, addItem, updateQuantity, ... }` |

项目中没有使用任何 `extends` 关键字。虽然未使用继承（也就无法演示 CARP 中"组合优于继承"的对比），但整体架构天然选择了组合方式，符合 CARP 的核心理念。

#### 正面案例：前端 `api.js` 的 HTTP 方法组合

[frontend/js/api.js:34-67](frontend/js/api.js#L34-L67) 中，`get`、`post`、`put`、`del` 都复用了 `_request` 通用方法：

```js
async _request(method, url, data = null) {
    const opts = { method, headers: this._headers() };
    if (data && (method === 'POST' || method === 'PUT' || method === 'PATCH')) {
      opts.body = JSON.stringify(data);
    }
    try {
      const res = await fetch(this.BASE_URL + url, opts);
      const json = await res.json();
      // 统一响应处理...
      return json;
    } catch (err) { ... }
},

get(url) { return this._request('GET', url); },
post(url, data) { return this._request('POST', url, data); },
put(url, data) { return this._request('PUT', url, data); },
del(url) { return this._request('DELETE', url); },
```

每个具体的 HTTP 方法**组合**了 `_request` 的核心逻辑，而不是通过继承获得 fetch 能力。这正是 CARP 的体现——组合让每个方法轻量且独立。

## 总结

| 设计原则 | 核心思想 | 项目优点 | 待改进之处 |
|---|---|---|---|
| **SRP** | 一个模块只做一件事 | 路由/控制器/模型分层清晰；`response.js` 职责专一 | `userController.updateProfile` 越权操作数据库；前端 `api.js` 过于庞大 |
| **OCP** | 对扩展开放，对修改关闭 | 中间件链天然支持扩展；`authorizeRoles` 参数化设计 | 订单模型状态转换硬编码，增加状态需改多处 |
| **LSP** | 子类型必须能替换父类型 | 所有中间件和控制器遵守统一函数签名 | 未利用 LSP 优势建立基类统一异常处理 |
| **LoD** | 只和直接朋友通信 | 控制器只依赖模型+响应工具；无链式调用 | `createProduct` 同时操作 Product 和 Audit 两个模型 |
| **DIP** | 依赖抽象而非具体实现 | 控制器依赖模型函数接口；路由依赖控制器接口 | 模型层直接依赖 `mysql2`；缺少 Repository 抽象 |
| **CARP** | 优先组合而非继承 | 中间件链是经典组合模式；模型模块零继承、独立复用 | 可提取共享错误处理 wrapper，进一步复用通用逻辑 |

总体而言，本项目通过 Express 框架的分层架构，在中间件链、路由与控制器的分离、模型模块化等方面较好地践行了六大设计原则。主要改进方向集中在：

1. **控制器中少量绕过模型层的直接数据库操作**（违反 SRP、DIP、LoD）
2. **订单状态转换缺乏扩展性设计**（违反 OCP）
3. **模型层对具体数据库的硬依赖**（违反 DIP）
4. **前端 API 模块的单体设计**（违反 SRP）

这些问题的修正成本不高，但能显著提升系统的可维护性和可扩展性。
