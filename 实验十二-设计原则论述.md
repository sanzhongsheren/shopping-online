# 实验十二 任务3：面向对象设计原则论述

> **项目**：商品网上交易系统（shopping-online）
> **技术栈**：Node.js + Express + MySQL
> **论述原则**：里氏代换原则、单一职责原则、开闭原则、迪米特法则、依赖倒转原则、合成复用原则

---

## 一、单一职责原则（Single Responsibility Principle, SRP）

### 定义

一个类或模块应该**有且仅有一个引起它变化的原因**。换句话说，一个模块只负责完成一项职责。

### 在项目中的体现

#### `utils/response.js` —— 专一的响应格式化工具

[backend/utils/response.js](backend/utils/response.js) 的职责极为单一——只负责统一 API 响应格式，所有控制器都通过它返回标准化 JSON：

```js
module.exports = {
    success(res, data = null, message = "success", code = 200) {
      return res.status(code).json({ code, message, data });
    },
    error(res, message = "服务器内部错误", code = 500) {
      return res.status(code).json({ code, message });
    },
    notFound(res, message = "资源不存在") {
      return res.status(404).json({ code: 404, message });
    },
    badRequest(res, message = "请求参数错误") {
      return res.status(400).json({ code: 400, message });
    }
};
```

当需要调整响应格式时，只需修改此文件，无需改动任何控制器。这正是 SRP 的核心价值——隔离变化的影响范围，每个模块只有一个理由去修改它。

#### 中间件职责分离

[backend/middleware/auth.js](backend/middleware/auth.js) 将认证功能拆分为两个独立的中间件：

- `authenticate`：**只负责**验证 JWT Token 并将用户信息挂载到 `req.user`
- `authorizeRoles`：**只负责**检查用户角色是否满足权限要求

两者职责清晰、互不干扰，可以在路由中按需独立使用或组合使用。

#### 路由与控制器分离

在 [backend/routes/orders.js](backend/routes/orders.js) 中，路由文件**仅负责 URL 到控制器的映射**，所有业务逻辑委托给控制器：

```js
router.post("/", authenticate, authorizeRoles("buyer"), orderController.createOrder);
router.get("/my", authenticate, authorizeRoles("buyer"), orderController.getMyOrders);
```

路由层不了解订单如何创建，控制器层不了解 HTTP 路由如何注册。各层职责分明。

## 二、开闭原则（Open/Closed Principle, OCP）

### 定义

软件实体（模块、类、函数）应该**对扩展开放，对修改关闭**。即新增功能应通过扩展实现，而非修改已有代码。

### 在项目中的体现

#### Express 中间件链的无限扩展能力

在 [backend/app.js](backend/app.js) 中，全局中间件通过 `app.use()` 层层叠加，构成了开放的请求处理管道：

```js
app.use(cors());
app.use(express.json());
app.use(express.urlencoded({ extended: true }));
app.use("/api/products", require("./routes/products"));
app.use("/api/users", require("./routes/users"));
app.use("/api/orders", require("./routes/orders"));
app.use("/api/audit", require("./routes/audit"));
app.use("/api/cart", require("./routes/cart"));
app.use("/api/messages", require("./routes/messages"));
app.use("/api/favorites", require("./routes/favorites"));
```

若要添加日志记录、限流、压缩等新功能，只需插入新的 `app.use(...)`，无需修改任何已有中间件或路由代码。若要新增一个业务模块（如"评价系统"），只需新建路由文件并添加一行挂载代码。

#### `authorizeRoles` 对角色扩展开放

[backend/middleware/auth.js:22-29](backend/middleware/auth.js#L22-L29) 使用 rest 参数接受任意数量的角色：

```js
exports.authorizeRoles = (...roles) => {
  return (req, res, next) => {
    if (!req.user || !roles.includes(req.user.role)) {
      return res.status(403).json({ code: 403, message: "权限不足" });
    }
    next();
  };
};
```

未来引入 `superadmin`、`moderator` 等新角色时，此函数本身无需任何修改——它在设计之初就对所有可能的角色扩展开放。使用时只需传入新角色名：`authorizeRoles("superadmin")`。

#### 路由层面的 OCP 实践

每条路由的中间件链同样是开放的。以 [backend/routes/orders.js](backend/routes/orders.js) 为例：

```js
router.post("/", authenticate, authorizeRoles("buyer"), orderController.createOrder);
```

若未来需要为创建订单添加输入校验、限流等新功能，只需在中间件链中插入新的中间件函数，无需修改 `authenticate`、`authorizeRoles` 或 `createOrder` 的任何代码。

## 三、里氏代换原则（Liskov Substitution Principle, LSP）

### 定义

子类型必须能够**完全替代**它们的父类型，而程序的行为不发生改变。即：程序中使用父类引用的地方，替换为子类对象后应依然正确运行。

### 在项目中的体现

本项目采用 JavaScript 函数式风格，未使用传统的类继承，但 LSP 的思想在**接口契约**层面得到了充分体现。

#### 中间件函数的可替换性

[backend/middleware/auth.js](backend/middleware/auth.js) 中，`authenticate` 和 `authorizeRoles()` 的返回值具有完全一致的函数签名 `(req, res, next) => void`：

```js
exports.authenticate = (req, res, next) => {
  const authHeader = req.headers.authorization;
  // 验证 token，设置 req.user
  next();
};

exports.authorizeRoles = (...roles) => {
  return (req, res, next) => {
    if (!req.user || !roles.includes(req.user.role)) {
      return res.status(403).json({ code: 403, message: "权限不足" });
    }
    next();
  };
};
```

由于两者具有相同的签名和语义（在特定条件下调用 `next()` 继续传递请求），它们在路由链中可以任意替换、调整顺序，程序行为依然正确：

```js
// 买家路由
router.post("/", authenticate, authorizeRoles("buyer"), orderController.createOrder);
// 卖家路由 —— 仅替换角色参数，行为完全一致
router.put("/:id/ship", authenticate, authorizeRoles("seller"), orderController.shipOrder);
```

这正是 LSP 在函数式编程中的实践——遵循相同契约的函数可以互相替换，无需修改调用方代码。

#### 控制器函数的统一接口

项目中所有控制器函数都遵循 `(req, res) => Promise<void>` 的签名约定。Express 路由不关心具体是哪个控制器的哪个函数——只要签名一致，就可以挂载到任意路由上。例如，`orderController.createOrder`、`productController.approveProduct`、`cartController.getCart` 对外呈现完全一致的接口形态，路由层对它们一视同仁，任意替换不会破坏程序运行。

## 四、迪米特法则（Law of Demeter, LoD）

### 定义

一个对象应该**对其他对象有尽可能少的了解**，只与"直接朋友"通信。也称为最少知识原则。通俗地说：**不要和陌生人说话**。

### 在项目中的体现

#### 控制器只与直接依赖通信

[backend/controllers/orderController.js](backend/controllers/orderController.js) 的依赖关系简洁明了：

```js
const Order = require("../models/order");
const { success, error, badRequest, notFound } = require("../utils/response");

exports.getMyOrders = async (req, res) => {
  try {
    const orders = await Order.getOrdersByBuyer(req.user.userId);
    success(res, { orders }, "订单列表获取成功");
  } catch (err) {
    console.error("获取订单列表失败:", err);
    error(res, "获取订单列表失败");
  }
};
```

控制器的"朋友"只有两个：
- `Order` 模型：提供数据访问能力
- `response` 工具：提供响应格式化能力

控制器不知道 `Order.getOrdersByBuyer()` 内部是调用 MySQL 还是其他数据库，不知道 SQL 语句长什么样，不知道数据库表结构。它只和"直接朋友"打交道。**整个项目中不存在 `a.b.c.d` 这种跨越多层的链式调用**。

#### 模型只与数据库连接通信

[backend/models/order.js](backend/models/order.js) 的依赖同样极其克制：

```js
const db = require("../config/db");
```

模型层只知道 `db` 这个数据库连接对象，不依赖其他模型、不依赖控制器、不依赖路由、不依赖请求/响应对象。"知识"范围被严格限定在数据持久化这一层。

#### 中间件的独立性

每个中间件只关心自己职责范围内的数据。`authenticate` 只关心请求头中的 Token，`authorizeRoles` 只关心 `req.user.role`。它们不关心请求体内容、不关心业务逻辑、不关心响应格式。这种"各管各的"设计正是迪米特法则的直观体现。

## 五、依赖倒转原则（Dependency Inversion Principle, DIP）

### 定义

- **高层模块不应依赖低层模块**，两者都应依赖抽象（接口）
- **抽象不应依赖细节**，细节应依赖抽象

### 在项目中的体现

#### 控制器依赖模型接口而非数据库实现

[backend/controllers/orderController.js](backend/controllers/orderController.js) 展示了典型的高层模块依赖抽象：

```js
const Order = require("../models/order");

exports.createOrder = async (req, res) => {
  try {
    const { addressId, paymentMethod, items } = req.body;
    const result = await Order.createOrder(req.user.userId, addressId, paymentMethod, items);
    success(res, result, "订单创建成功");
  } catch (err) {
    console.error("创建订单失败:", err);
    error(res, err.message || "创建订单失败");
  }
};
```

控制器（高层模块）依赖的是 `Order.createOrder()` 这个**函数接口**，而非低层的 MySQL 连接、SQL 语句或事务处理细节。Order 模块对外暴露的是一组语义清晰的函数签名，隐藏了所有实现细节。这正是"面向接口编程"的实践。

#### 路由依赖控制器接口

[backend/routes/orders.js](backend/routes/orders.js) 作为框架层，只依赖控制器的函数签名：

```js
const orderController = require("../controllers/orderController");
router.post("/", authenticate, authorizeRoles("buyer"), orderController.createOrder);
```

路由只知道 `orderController.createOrder` 是一个符合 `(req, res) => Promise<void>` 签名的函数。它与控制器内部实现完全解耦——无论控制器内部如何重构（改用 Service 层、引入事务管理、更换数据库），路由代码一行不用改。

#### 分层架构本身就是 DIP 的体现

项目的整体架构形成了一条清晰的依赖链，每一层都依赖下一层暴露的**抽象接口**而非具体实现：

```
路由层 → 依赖控制器函数签名
控制器层 → 依赖模型函数签名
模型层 → 依赖数据库连接对象
```

每一层都可以独立演进和替换，只要保持接口契约不变。例如可以重写整个 `order.js` 模型改用 ORM（如 Sequelize），只要导出的函数名和参数不变，控制器代码无需任何修改。

## 六、合成复用原则（Composite/Aggregate Reuse Principle, CARP）

### 定义

**优先使用对象组合（composition）而非类继承（inheritance）来实现复用**。继承是静态的、编译期确定的，而组合是动态的、运行时可替换的，更加灵活。

### 在项目中的体现

#### Express 中间件链 —— 组合的典范

[backend/routes/orders.js](backend/routes/orders.js) 中，每个路由的行为由多个独立的中间件**水平组合**而成：

```js
// 买家创建订单：组合 认证 + 买家权限 + 业务处理
router.post("/", authenticate, authorizeRoles("buyer"), orderController.createOrder);

// 卖家发货：组合 认证 + 卖家权限 + 业务处理
router.put("/:id/ship", authenticate, authorizeRoles("seller"), orderController.shipOrder);

// 确认收货：组合 认证 + 买家权限 + 业务处理
router.put("/:id/receive", authenticate, authorizeRoles("buyer"), orderController.confirmReceive);

// 取消订单：组合 认证 + 业务处理（买卖家均可）
router.put("/:id/cancel", authenticate, orderController.cancelOrder);
```

不同路由通过组合不同的中间件来产生不同的权限行为。这比继承方式灵活得多——假设用继承来设计，需要创建 `BuyerRoute`、`SellerRoute`、`AdminRoute` 等子类，而组合方式可以在运行时**任意编排**中间件顺序和组合，无需创建任何子类。

#### 高阶函数模式 —— `authorizeRoles` 的参数化组合

[backend/middleware/auth.js:22](backend/middleware/auth.js#L22) 使用工厂函数（高阶函数）模式：

```js
exports.authorizeRoles = (...roles) => {
  return (req, res, next) => {
    if (!req.user || !roles.includes(req.user.role)) {
      return res.status(403).json({ code: 403, message: "权限不足" });
    }
    next();
  };
};
```

`authorizeRoles("seller")` 和 `authorizeRoles("admin")` 的差异仅由传入的参数决定，**复用同一段核心逻辑**，无需为每种角色编写独立的中间件。这比继承方案（`class SellerMiddleware extends RoleMiddleware`）更简洁、更灵活。

#### 模型模块独立，零继承，天然组合

所有模型文件均为独立的 Node.js 模块，导出纯函数，不存在 `BaseModel` 父类：

| 模型文件 | 导出接口 |
|---|---|
| [backend/models/user.js](backend/models/user.js) | `{ createUser, findByUsername, findById }` |
| [backend/models/product.js](backend/models/product.js) | `{ getProductList, getProductById, createProduct, ... }` |
| [backend/models/order.js](backend/models/order.js) | `{ createOrder, getOrdersByBuyer, payOrder, shipOrder, ... }` |
| [backend/models/cart.js](backend/models/cart.js) | `{ getCartByUserId, addItem, updateQuantity, ... }` |

项目中没有使用任何 `extends` 关键字，每个模型都是一个自包含的模块，通过 `require()` 进行组合使用。控制器需要哪些模型功能，就引入哪些——按需组合，而非通过继承链获得一堆不需要的方法。

#### 前端 API 模块的 HTTP 方法组合

[frontend/js/api.js:34-67](frontend/js/api.js#L34-L67) 中，`get`、`post`、`put`、`del` 全部复用了 `_request` 通用方法：

```js
async _request(method, url, data = null) {
    const opts = { method, headers: this._headers() };
    if (data && (method === 'POST' || method === 'PUT' || method === 'PATCH')) {
      opts.body = JSON.stringify(data);
    }
    const res = await fetch(this.BASE_URL + url, opts);
    const json = await res.json();
    return json;
},

get(url) { return this._request('GET', url); },
post(url, data) { return this._request('POST', url, data); },
put(url, data) { return this._request('PUT', url, data); },
del(url) { return this._request('DELETE', url); },
```

每个 HTTP 方法通过**组合**共享 `_request` 的核心能力，而非通过继承获取。如果需要新增 `patch` 方法，只需添加一行组合调用——完全符合合成复用原则。

## 总结

| 设计原则 | 核心思想 | 在本项目中的体现 |
|---|---|---|
| **SRP** 单一职责 | 一个模块只做一件事 | `response.js` 专一负责响应格式；路由/控制器/模型三层各司其职；认证与授权中间件分离 |
| **OCP** 开闭原则 | 对扩展开放，对修改关闭 | 中间件链可无限扩展；`authorizeRoles(...)` 接受任意角色参数；新增路由模块只需一行挂载 |
| **LSP** 里氏代换 | 子类型必须能替换父类型 | 所有中间件共享 `(req,res,next)` 签名可任意替换；控制器函数接口统一 |
| **LoD** 迪米特法则 | 只和直接朋友通信 | 控制器只依赖模型+响应工具；模型只依赖 `db`；项目无链式调用 |
| **DIP** 依赖倒转 | 依赖抽象而非具体实现 | 控制器依赖模型函数接口；路由依赖控制器函数接口；分层架构天然倒置依赖 |
| **CARP** 合成复用 | 优先组合而非继承 | 中间件链水平组合行为；高阶函数模式；模型零继承独立复用；HTTP 方法组合 `_request` |

本项目通过 Express 框架的分层架构和中间件模式，在六大设计原则方面均有较好的实践。中间件链的组合模式同时体现了 OCP、LSP、CARP 三个原则的精髓；三层架构的依赖方向天然符合 DIP；模块化的路由/控制器/模型拆分使 SRP 和 LoD 得以落实。这些设计原则不是孤立的教条，而是在实际编码中相互交织、互相支撑的工程智慧。
