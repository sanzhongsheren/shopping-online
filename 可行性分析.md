## 1 引言

### 1.1 标识

本文档适用于**商品网上交易系统**课程开发项目，版本号V1.0，为项目初期可行性分析初稿，覆盖系统全功能模块的可行性论证。

### 1.2 背景

本项目为软件工程课程实践项目，针对线下商品交易信息不对称、交易流程不规范、买卖双方对接效率低的问题，提出开发一套轻量化的网上商品交易系统，实现用户注册登录、商品发布与审核、在线下单、订单状态全流程跟踪的闭环交易能力。
项目开发方为在校学生团队，使用方为普通买家、卖家及系统管理员，运行环境为PC端浏览器。

### 1.3 项目概述

本项目旨在开发一套前后端分离的B/S架构商品网上交易系统，核心覆盖**用户管理、商品管理、审查管理、购物下单、订单配送状态管理**五大核心模块，实现商品交易的全流程线上化。
项目周期为课程学期内，开发完成后将完成功能测试、部署上线，作为课程项目成果交付。

### 1.4 文档概述

本文档用于论证项目开发的可行性，明确项目需求、目标、可选方案，从技术、经济、法律、用户使用四个核心维度分析项目落地的可能性与风险，文档内容仅用于课程项目开发，无保密要求。

## 2 可行性分析的前提

### 2.1 项目的核心要求

系统必须实现以下核心功能要求：

1. **用户管理**：支持买家/卖家用户的注册、登录、留言功能；
2. **商品管理**：支持卖家发布带图片的商品信息，可对自有商品进行增删改查管理；
3. **审查管理**：管理员可对用户注册资质、卖家发布的商品信息进行审核，审核通过后方可生效；
4. **购物下单**：买家可检索、查看审核通过的商品，支持在线下单、生成订单；
5. **订单配送管理**：支持订单状态全流程流转，卖家可标记订单为“已出货”，买家收货后可标记为“已收货”，完成交易闭环。

### 2.2 项目的目标

1. 功能目标：完成五大核心模块的开发，实现商品交易全流程线上化，功能完整、流程闭环；
2. 性能目标：支持多用户同时操作，页面响应流畅，数据提交、状态更新无明显延迟；
3. 易用性目标：界面简洁直观，买家、卖家、管理员三类角色操作门槛低，无需额外培训即可上手；
4. 项目成果目标：完成符合软件工程规范的课程项目，交付可运行的系统及配套设计文档。

### 2.3 项目的环境、假定和限制

1. 开发环境：基于主流的前后端开发技术栈，开发设备为普通PC机，部署环境可使用本地服务器、云服务器免费额度或课程提供的部署环境；
2. 项目周期：限制在课程学期内，开发周期约2-3个月；
3. 开发人员：学生开发团队，具备基础的前后端开发、数据库设计能力；
4. 成本限制：无额外商业经费投入，优先使用开源免费的技术、组件、部署资源；
5. 功能边界：聚焦于商品交易核心流程，不实现复杂的在线支付、物流对接、售后维权等扩展功能。

### 2.4 可行性分析的方法

本次分析采用以下方法：

1. 需求分析法：明确核心功能与非功能需求，锚定项目开发边界；
2. 技术对比法：对比可选的技术方案，评估开发难度与适配性；
3. 成本效益分析法：评估项目开发的资源投入与产出效益；
4. 风险评估法：预判开发过程中的技术、进度风险，提出应对思路。

## 3 可选方案分析

### 3.1 原有方案的局限性

线下商品交易/纯人工对接的原有模式，存在核心问题：

1. 信息不对称：买家无法快速获取商品信息，卖家的商品展示渠道有限；
2. 流程不规范：订单状态、交易进度无统一记录，易出现纠纷；
3. 效率低下：从商品咨询到下单、收货，全流程人工对接，耗时耗力；
4. 无审核机制：商品信息、用户资质无审核，存在虚假信息、违规交易的风险。

### 3.2 可选择的系统方案

#### 方案1：前后端分离轻量化方案（推荐）

- 技术选型：前端采用Vue/React轻量化框架，后端采用SpringBoot/Node.js/Python Flask，数据库采用MySQL，实现前后端解耦，开发效率高，适配课程项目的快速迭代；
- 优势：技术栈成熟、开源资料丰富、学习成本低，能快速实现核心功能，符合学生开发团队的能力范围；
- 局限性：高并发能力有限，仅适配小体量用户使用，满足课程项目需求完全足够。

#### 方案2：单体应用快速开发方案

- 技术选型：采用PHP等脚本语言，基于现成的商城模板二次开发，快速搭建功能；
- 优势：开发速度快，无需从零搭建基础框架；
- 局限性：自定义修改难度大，不利于理解软件工程全流程，不符合课程项目的学习目标，代码规范性、可维护性差。

### 3.3 最终方案选择准则

1. 符合课程项目的学习目标，能完整实践软件工程的需求分析、设计、开发、测试全流程；
2. 适配开发团队的技术能力，学习成本低，开源资料丰富，能规避核心技术风险；
3. 能100%覆盖项目核心功能需求，开发周期可控，能在学期内完成交付；
4. 无额外成本投入，全部采用开源免费的技术与资源。

基于以上准则，**优先选择方案1（前后端分离轻量化方案）** 作为项目最终开发方案。

## 4 所建议的系统方案

### 4.1 系统说明

本系统采用B/S架构，基于前后端分离模式开发，分为买家、卖家、管理员三类角色，每个角色对应不同的功能权限，围绕“商品发布-审核-下单-订单流转-交易完成”的核心流程，实现五大功能模块的完整闭环。

### 4.2 核心业务流程

1. 用户流程：用户注册→管理员审核资质→登录系统→买家/卖家角色功能使用；
2. 商品流程：卖家发布带图片的商品信息→管理员审核商品→审核通过后商品在前台展示；
3. 交易流程：买家检索商品→下单生成订单→卖家确认出货→买家确认收货→交易完成。

### 4.3 影响与要求

1. **设备**：开发与运行仅需普通PC机、浏览器，无需特殊硬件设备；
2. **软件**：采用开源免费的开发工具、数据库、框架，无需商业软件授权；
3. **开发**：需按照软件工程规范，完成需求分析、设计、编码、测试全流程，产出配套文档；
4. **运行**：系统部署后，可在主流浏览器中访问使用，无需用户安装额外客户端。

### 4.4 局限性

本系统为课程实践项目，存在以下局限性：

1. 未对接第三方在线支付接口，仅实现订单状态流转，不处理真实资金交易；
2. 未对接第三方物流系统，物流状态需卖家手动更新，无实时物流跟踪能力；
3. 未设计复杂的售后、维权、评价体系，聚焦核心交易流程；
4. 性能设计仅适配小体量用户，不支持高并发、大流量的商业级场景。

## 5 经济可行性分析

本项目为高校课程实践项目，无商业盈利目标，核心从成本与效益两方面分析：

### 5.1 成本投入

项目全程无额外资金投入，所有成本为人力与时间成本：

1. 开发资源：全部采用开源免费的技术栈、开发工具、数据库，无软件授权、硬件采购费用；
2. 部署成本：可使用本地环境运行，或云厂商免费额度的服务器进行部署，无服务器租赁费用；
3. 人力成本：开发团队为在校学生，利用课程学习与课余时间完成开发，无人力费用支出。

### 5.2 预期效益

1. 学习效益：完整实践软件工程的全流程开发，掌握前后端开发、数据库设计、项目管理的核心能力，完成课程学习目标；
2. 成果效益：产出完整可运行的系统项目，可作为个人作品集、课程成果，用于后续学习与实践参考；
3. 实践效益：理解电商交易系统的核心业务逻辑，为后续的进阶开发、商业项目实践积累经验。

### 5.3 结论

项目无资金投入风险，仅需投入合理的时间与人力，即可完成开发并获得显著的学习与实践收益，**经济层面完全可行**。

## 6 技术可行性分析

### 6.1 核心技术能力评估

1. **技术栈成熟度**：建议采用的前后端分离技术栈（如Vue+SpringBoot+MySQL）是目前业界最主流的web开发技术，开源社区成熟，教程、解决方案丰富，遇到技术问题可快速找到解决方法；
2. **开发团队能力**：开发人员为软件工程相关专业学生，具备编程语言、数据库、web开发的基础能力，通过课程学习与资料查阅，可快速掌握所需技术，满足开发需求；
3. **功能实现可行性**：项目核心的用户管理、商品管理、审核、订单管理功能，均为web开发的经典场景，无技术壁垒，图片上传、状态更新等功能均有成熟的实现方案，可稳定落地。

### 6.2 技术风险与应对

1. 风险：开发人员对部分技术不熟悉，导致开发进度延迟；
   应对：提前进行技术预研，拆分功能模块，分工开发，团队内共享技术经验，优先实现核心功能。
2. 风险：部署、环境配置出现问题，导致系统无法正常运行；
   应对：采用轻量化的部署方案，提前完成环境搭建与测试，使用成熟的部署工具降低操作难度。

### 6.3 结论

项目所需技术均为成熟开源技术，无核心技术壁垒，适配开发团队的能力范围，开发周期可控，**技术层面完全可行**。

## 7 法律可行性分析

1. **知识产权**：项目全部采用开源免费的技术与框架，遵守对应的开源协议，代码为团队自主开发，无侵权风险；
2. **数据合规**：项目仅用于课程实践，不收集用户的敏感隐私信息，不对外泄露、售卖用户数据，符合个人信息保护的相关要求；
3. **内容合规**：系统设置管理员审核机制，可对用户发布的商品信息、留言内容进行审核，规避违规、违法内容发布的风险；
4. **使用边界**：项目仅用于课程学习与实践，不用于商业运营，无经营性相关的法律风险。

结论：项目开发与使用过程中，可有效规避法律风险，**法律层面完全可行**。
